What is Symbol tables

Symbol Tables

Key-value pair abstraction

  • $ \color{red}{Insert} $ a value with specified key.
  • Given a key, $ \color{red}{search} $ for the corresponding value.

Basic symbol table API

Conventions

  • Values are not null.
  • Method get() returns null if key not present.
  • Method put() overwrites old value with new value.

Intended consequences

  • Easy to implement contains()
  • Can implement lazy version of delete()

Keys and values

Value type: Any generic type.
Key type: several natural assumptions.

  • Assume keys are Comparable, use compareTo().
  • Assume keys are any generic type, use equals() to test equality.
  • Assume keys are any generic type, use hashcode() to scramble key.
    Best practices: Use immutable types for symbol table keys.
  • Immutable in Java: String, Integer, Double, java.io.File…
  • Mutable in Java: StringBuilder, java.net.URL, arrays, …

Equality test

All java classes inherit a method equals().
Java requirements. For any references x, y and z:

  • Reflexive: x.equals(x) is true.
  • Symmetric: x.equals(y) iff y.equals(x).
  • Transitive: if x.equals(y) and y.equals(z), then x.equals(z).
  • Non-null: x.equals(null) is false.

Default implementation. (x == y)

  • Do x and y refer to the same object?

Customized implementations.

  • Integer, Double, String, File, URL,…

User-defined implementations.

  • Some care needed.

Implementing equals for user-defined types

Equals design
“Standard” recipe for user-defined types.

  • Optimization for reference equality.
  • Check against null.
  • Check that two objects are of the same type and cast.
  • Compare each significant field:
    – if field is a primitive type, use ==
    – if field is an object, use equals() (apply rule recursively)
    – if field is an array, apply to each entry (alternatively, use Arrays.equals(a,b) or Arrays.deepEquals(a, b), but not a.equals(b))

Best practices.

  • No need to use calculated fields that depend on other fields.
  • Compare fields mostly likely to differ first.
  • Make compareTo() consistent with equals(). (x.equals(y) if and only if (x.compareTo(y) == 0))

Perceptron Hypothesis Set

A Simple Hypothesis Set: the ‘Perceptron’

  • For $ \small{x = (x_1,x_2,…,x_d)} $ ‘feature of customer’, compute a weighted ‘score’ and:
    approve credit if $ \small{\sum\limits_{i = 1}^dw_ix_i} > $ threshold
    deny credit if $ \small{\sum\limits_{i = 1}^dw_ix_i} < $ threshold
  • {+1(good), -1(bad)}, 0 ignored - linear formula $ \small{h \in H} $ are
    $ \small{\color{red}{h}(x) = sign((\sum\limits_{i=1}^{d}\color{red}{w_i}x_i) - \color{red}{threshold})} $

called ‘perceptron’ hypothesis historically

Vector Form of Perceptron Hypothesis

  • $ \small{\color{red}{h}(x) = sign((\sum\limits_{i=1}^{d}\color{red}{w_i}x_i) - \color{blue}{threshold})} $

$ \small{\qquad= sign((\sum\limits_{i=1}^{d}\color{red}{w_i}x_i) + \color{blue}{(-threshold) * (+1)} )} $

$ \small{\qquad = sign(\sum\limits_\color{blue}{i=0}^{d}\color{red}{w_i}x_i)} $

$ \small{\qquad = sign(\color{red}{w^T}x)} $

  • each ‘tall’ w represents a hypothesis h & is multiplied with ‘tall’ x(will use tall versions to simplify notation)

Perceptrons in $ \small{R^2} $

perceptrons $ \Leftrightarrow $ linear(binary) classifiers

Perceptrons Learning Algorithm

Select g from H

H = all possible perceptrons, g = ?

  • want: $ \small{g \approx f} $ (hard when $ \small{f} $ unknown)
  • almost necessary: $ \small{g \approx f} $ on $ \small{D} $ , ideally $ \small{\color{purple}{g(x_n) = f(x_n) = y_n}} $
  • difficult: H is of infinite size
  • idea: start from some $ \small{g_0} $, and ‘correct’ its mistakes on $ \small{D} $

will represent $ \small{g_0} $ by its weight vector $ w_0 $

Perceptron Learning Algorithm

start from some $ \small{w_0} $ (say, 0), and correct its mistakes on $ \small{D} $

next can follow naive cycle $ \small{1,…N} $ or precomputed random cycle

Some Remaining Issues of PLA

‘correct’ mistakes on D until no mistakes

Guarantee of PLA

Linear Separability

  • if PLA halts (i.e. no more mistakes), $ \small{D} $ allows some $ \small{w} $ to make no mistake
  • call such $ \small{D} $ linear separable

PLA fact: $ \small{w_t} $ Gets More Aligned with $ \small{w_f} $

linear seperable $ \small{D} \Leftrightarrow $ exists perfect $ w_f $ such that
$ \small{y_n = sign(w_f^Tx_n)} $

  • $ \small{w_f} $ perfect hence every $ \small{x_n} $ correctly away from line: $ \small{ y_{n(t)}w_f^Tx_{n(t)} \geq \min\limits_n y_nw_f^Tx_n > 0} $
  • $ \small{w_f^Tw_t\uparrow} $ by updating with any $ \small{(x_{n(t)}, y_{n(t)})} $ :
    $ \small{w_f^Tw_{t+1} = w_f^T(w_t + y_{n(t)}x_{n(t)})} $

$ \qquad\quad \small{\geq w_f^Tw_t + \min\limits_n y_nw_f^Tx_n} $

$ \qquad\quad \small{> w_f^Tw_t + 0} $

PLA fact: $ \small{w_t} $ Does Not Grow Too Fast

$ \small{w_t} $ changed only when mistake
$ \Leftrightarrow \small{sign(w_t^Tx_{n(t)}) \neq y_{n(t)} \Leftrightarrow y_{n(t)}w_t^Tx_{n(t)} \leq 0} $

  • mistake ‘limits’ $ \Vert w_t\Vert^2 $ growth,
    $ \small{\Vert w_{t+1}\Vert^2 = \Vert w_t + y_{n(t)}x_{n(t)}\Vert^2} $

$ \small{\qquad\quad = \Vert w_t\Vert^2 + 2y_{n(t)}w_t^Tx_{n(t)} + \Vert y_{n(t)}x_{n(t)}\Vert^2} $

$ \small{\qquad\quad \leq \Vert w_t\Vert^2 + 0 + \Vert y_{n(t)}x_{n(t)}\Vert^2} $

$ \small{\qquad\quad \leq \Vert w_t\Vert^2 + 0 + \max\limits_n\Vert y_nx_n\Vert^2} $

start from $ \small{w_0 = 0} $, after $ \small{T} $ mistake corrections,
$ \frac{w_f^T}{\Vert w_f\Vert}\frac{w_T}{\Vert w_T\Vert} \small{\geq \sqrt{T} \cdot } $ constant

proof see here

Therefore, $ \small{T} $ have upper bound.

More about PLA

Guarantee
as long as linear separable and correct by mistake

  • inner product of $ \small{w_f} $ and $ \small{w_t} $ grows fast; length of $ \small{w_t} $ grows slowly
  • PLA ‘lines’ are more and more aligned with $ \small{w_f} \Rightarrow $ halts

Pros
simple to implement, fast, works in any dimension d

Cons

  • ‘assumes’ linear separable D to halt
    – properly unknown in advance (no need for PLA if we know $ \small{w_f} $)
  • not fully sure how long halting takes ($ \rho $ depends on $ \small{w_f} $)
    – though practically fast

Binary heaps

Complete binary tree

Binary tree: Empty or node with links to left and right binary trees.
Complete tree: Perfectly balanced, except for bottom level.
Property: Height of complete tree with N nodes is [$ \small{lgN} $].
Pf. Height only increases when N is a power of 2.

Binary heap representations

Binary heap: Array representation of a heap-ordered complete binary tree.

Heap-ordered binary tree

  • Keys in nodes.
  • Parent’s key no smaller than children’s keys.

Array representation

  • Indices start at 1.
  • Take nodes in level order.
  • No explicit links needed!

Binary heap properties

Proposition:

  • Largest key is a[1], which is root of binary tree.
  • Can use array indices to move through tree.
    – Parent of node at k is at k/2.
    – Children of node at k is at 2k and 2k+1.

Promotion in a heap

Scenario: Child’s key becomes larger than its parent’s key.

To eliminate the violation:

  • Exchange key in child with key in parent.
  • Repeat until heap order restored.

Insertion in a heap

Insert: Add node at end, then swim it up.
Cost: At most $ \small{1 + lgN} $ compares.

Demotion in a heap

Scenario: Parent’s key becomes smaller than one (or both) of its children’s.

To eliminate the violation:

  • Exchange key in parent with key in larger child.
  • Repeat until heap order restored.

Delete the maximum in a heap

Delete Max: Exchange root with node at end, then sink it down.
Cost: At most $ \small{2lgN} $ compares.

Heapsort: mathematical analysis

Proposition:

  • Heap construction uses $ \small{\leq} $ $ \small{2N} $ compares and exchanges.
  • Heapsort uses $ \small{\leq} $ $ \small{2NlgN} $ compares and exchanges.

Significance: In-place sorting algorithm with $ \small{NlogN} $ worst-case.(inplace merge possible, not practical)

  • Mergesort: no, linear extra space.($ \small{NlogN} $ worst-case quicksort possible, not practical)
  • Quicksort: no, quadratic time in worst case.($ \small{NlogN} $ worst-case quicksort possible, not practical)
  • Heapsort: yes!

Bottom line: Heapsort is optimal for both time and space, but:

  • Inner loop longer than quicksort’s.(2 compares)
  • Makes poor use of cache memory.
  • Not stable.

Molecular dynamics simulation of hard discs

Goal: Simulate the motion of N moving particles that behave according to the laws of elastic collision.

Hard disc model

  • Moving particles interact via elastic collisions with each other and walls.
  • Each particle is a disc with known position, velocity, mass, and radius.
  • No other forces.

Warmup:bouncing balls
N bouncing balls in the unit square. (without checking balls colliding with each other)

Time-driven simulation

  • Discretize time in quanta of size dt.
  • Update the position of each particle after every dt units of time, and check for overlaps.
  • If overlap, roll back the clock to the time of the collision, update the velocities o the colliding particles, and continue the simulation.

Main drawbacks:

  • ~$ \small{N^2 / 2} $ overlap checks per time quantum.
  • Simulation is too slow if $ \small{dt} $ is very small.
  • May miss collisions if $ \small{dt} $ is too large.
    (if colliding particles fail to overlap when we are looking)

Event-driven simulation
Change state only when something happens:

  • Between collisions, particles move in straight-line trajectories.
  • Focus only on times when collisions occur.
  • Maintain PQ of collision events, prioritized by time.
  • Remove the min = get next collision.

Collision prediction: Given position, velocity, and radius of a particle, when will it collide next with a wall or another particle?

  • Particle $ \small{i} $: radius $ s_i $, position $ \small{(rx_i, ry_i)} $, velocity $ \small{(vx_i, vy_i)} $.
  • Particle $ \small{j} $: radius $ s_j $, position $ \small{(rx_j, ry_j)} $, velocity $ \small{(vx_j, vy_j)} $.
  • Will particles $ \small{i} $ and $ \small{j} $ collide? If so, when?

Collision resolution: If collision occurs, update colliding particles according to laws of elastic collisions.

Particle data type skeleton

Initialization:

  • Fill PQ with all potential particle-wall collisions.
  • Fill PQ with all potential partical-partical collisions.
    (potential since collision may not happen if some other collision intervenes)

Mainloop:

  • Delete the impending event from PQ(minpriority = t).
  • If the event has been invalidated, ignore it.
  • Advance all particles to time t, on a straight-line trajectory.
  • Update the velocities of the colliding particles.
  • Predict future particle-wall and particle-particle collisions involving the colliding particles and insert events onto PQ.

Basic plan

  • Shuffle the array.
  • Partition so that, for some j
    – entry a[j] is in place
    – no larger entry to the left of j
    – no smaller entry to the right of j
  • Sort each piece recursively

Process

Phase 1. Repeat until i and j pointers cross.

  • Scan i from left to right so long as (a[i] < a[lo]).
  • Scan j from right to left so long as (a[j] > a[lo]).
  • Exchange a[i] with a[j].

Phase 2. When pointers cross.

  • Exchange a[lo] with a[j].

Java Code

Implementation details

  • Partitioning inplace. Using an extra array makes partitioning easier(and stable), but is not worth the cost.

  • Terminating the loop. Testing whether the pointers cross is a bit trickier than it might seem.

  • Staying in bounds. The (j == lo) test is redundant, but the (i == hi) test is not.
    Because the definition of less:

  • Preserving randomness. Shuffling is needed for performance guarantee.

  • Equal keys. When duplicates are present, it is (counter-intuitively) better to stop on keys equal to the partitioning item’s key.

Performace analysis

  • Best case: Number of compares is ~$ \small{NlgN} $

  • Worst case: Number of compares is ~$ \small{\frac{1}{2}N^2}$

    • $ \small{N + (N-1) + (N-2) + … + 1} $ ~$ \small{\frac{1}{2}N^2}$
    • More likely that your computer is struck by lightning bolt.

  • Average case: The average number of compares $ \small{C_N}$ to quicksort an array of $ \small{N} $ distinct keys is ~$ \small{2NlnN}$ (and the number of exchanges is ~$ \small{\frac{1}{3}NlnN }$).

    • 39% more compares than mergesort.
    • But faster than mergesort in practice because of less data movement.

    Pf: $ \small{C_N} $ satisfies the recurrence $ \small{C_0 = C_1 = 0} $ and for $ \small{N \geq 2} $:
    $ \small{C_N = (N+1) + (\frac{C_0 +C_{N-1}}{N}) + (\frac{C_1 +C_{N-2}}{N}) + … + (\frac{C_{N-1} +C_{0}}{N})} $
    Multiply both sides by $ \small{N} $ and collect terms:
    $ \small{NC_N = N(N+1)+2(C_0 + C_1 + … + C_{N-1})} $
    Subtract this from the same equation for $ N-1 $:
    $ \small{NC_N - (N-1)C_{N-1} = 2N + 2C_{N-1}} $
    Rearrange terms and divide by $ \small{N(N+1)} $:
    $ \frac{C_N}{N+1} = \frac{C_{N-1}}{N} + \frac{2}{N+1} $
    Repeatedly apply above equation:
    $ \frac{C_N}{N+1} = \frac{C_{N-1}}{N} + \frac{2}{N+1} $
    $ \qquad= \frac{C_{N-2}}{N-1} + \frac{2}{N} + \frac{2}{N+1}$
    $ \qquad= \frac{2}{3} + \frac{2}{4} + \frac{2}{5} + … + \frac{2}{N+1}$
    Approximate sum by an integral:
    $ \small{C_N = 2(N+1)(\frac{1}{3} + \frac{1}{4} + \frac{1}{5} + … + \frac{1}{N+1})} $
    $ \small\quad $ ~ $ \small{2(N+1)\int_3^{N+1}\frac{1}{x}dx} $
    Finally, the desired result:
    $ \small{C_N} $ ~ $ \small{2(N+1)lnN \approx 1.39NlgN} $

3-way partitioning
Partition array into 3 parts so that:

  • Entries between lt and gt equal to partition item v.
  • No larger entries to left of lt.
  • No smaller entries to right of gt.

Sorting lower bound
If there are n distinct keys and the ithi^{th}ith one occurs xix_ixi​ times, any compare-based sorting algorithm must use at least
$ \small{lg(\frac{N!}{x_1!x_2!..x_n!})} $ ~ $ \small{-\sum\limits_{i=1}^{n}x_ilg\frac{x_i}{N}} $
compares in the worst case.

  • $ \small{NlgN} $ when all distinct;
  • linear when only a constant number of distinct keys.

Proposition.
Quicksort with 3-way partitioning is entropy-optimal.

Bottom line.
Ramdomized quicksort with 3-way partitioning reduces running time from linearithmic to linear in broad class of applications.

Java System sorts

Arrays.sort()

  • Has different method for each primitive type.
  • Has a method for data types that implement Comparable.
  • Has a method that uses a Comparator.
  • Uses tuned quicksort for primitive types; tuned mergesort for objects.

Why use different algorithms for primitive and reference types?
Because if the programmer using reference types, space may not be an important problem.

  • 修饰的函数不可以被覆盖。
  • 修饰的变量是一个常量，只能赋值一次。
  • 内部类定义在局部时，只能访问被 final 修饰的局部变量。（当然，因为还持有 Outer.this 引用，还可以访问外部类中的成员）

From Learning to Machine Learning

  • learning:acquiring skill
    with experience accumulated from observations
    observations — learning — skill

  • machine learning: acquiring skill
    with experience accumulated/ computed from data
    data — ML — skill

What is skill?

skill: improve some performance measure(e.g. prediction accuracy)

  • machine learning: improving some performance measure with experience computed from data
    data — ML — improved performace measure

An Application in Computational Finance

  • stock data — ML — more investment gain

Why use machine learning?

  • ‘define’ trees and hand-program: difficult
  • learn from data(observations) and recognize: a 3-year-old can do so
  • ‘ML-based tree recognition system’ can be easier to build than hand-programmed system

The Machine Learning Route

ML: an alternative route to build complicated systems

Some Use Scenarios:

  • when human cannot program the system manually
    – navigating on Mars
  • when human cannot ‘define the solution’ easily
    – speech/visual recognition
  • when needing rapid decisions that humans cannot do
    – high-frequency trading
  • when needing to be user-oriented in a massive scale
    – consumer-targeted marketing

Key Essence of Machine Learning

  • machine learning: improving some performance measure with experience computed from data
    data — ML — improved performace measure
  1. exists some underlying pattern to be learned
    – so ‘performance measure’ can be improved
  2. but no programmable (easy) definition
    – so ‘ML’ is needed
  3. somehow there is data about the pattern
    – so ML has some ‘inputs’ to learn from

key essence: help decide whether to use ML

Applications of Machine Learning

  • Food:(Sadilek et al.,2013)
    – data: Twitter data (words + location)
    – skill: tell food poisoning likeliness of restaurant properly
  • Clothing:(Abu-Mostafa, 2012)
    – data: sales figures + client surveys
    – skill: give good fashion recommendations to clients
  • Housing:(Tsanas and Xifara, 2012)
    – data: characteristics of buildings and their energy load
    – skill: predict energy load of other buildings closely
  • Transportation:(Stallkamp at al.,2012)
    – data: some traffic sign images and meanings
    – skill: recognize traffic signs accurately
  • Education
    – data: students’ records on quizzes on a Math tutoring system
    – skill: predict whether a student can give a correct answer to another quiz question
    A Possible ML Solution
    • give ML 9 million records from 3000 students
    • ML determines (reverse-engineers) strength and
      difficulty automatically
  • Entertainment: Recommender System
    – data: how many users have rated some movies
    – skill: predict how a user would rate an unrated movie
    A Hot Problem
    • competition held by Netflix in 2006
      – 100480507 ratings that 480189 users gave to 17770 movies
      – 10% improvement = 1 million dollar prize
    • similar competition (songs) held by Yahoo! in KDDCup 2011
      – 252800275 ratings that 1000990 users gave to 624961 songs
      A Possible ML Solution
    • pattern:
      rating <- viewer/movie factors
    • learning:
      known rating
      -> learned factors
      -> unknown rating prediction

Components of Learning: Metaphor Using Credict Approval

Basic Notations

  • input: $ \small{x \in \chi} $ (customer application)
  • output: $ \small{y \in Y} $ (good/bad after approving credit card)
  • unknown pattern to be learned $ \Leftrightarrow $ target function:
    $ \small{f: \chi \rightarrow Y} $ (ideal credit approval formula)
  • data $ \Leftrightarrow $ training examples:
    $ \small{D = {(x_1, y_1),(x_2, y_2),…,(x_N, y_N)}} $ (historical records in bank)
  • hypothesis $ \Leftrightarrow $ skill with hopefully good performance:
    $ \small{g: \chi \rightarrow Y} $ (‘learned’ formula to be used)

$ \small $ from $ \small{f} \rightarrow ML \rightarrow g $

Remark:

  • target $ f $ unknown (i.e. no programmable definition)
  • hypothesis $ g $ hopefully $ \small{\approx f} $
    but possibly different from f\small{f}f(perfection ‘impossible’ when $ \small{f} $ unknown)

  • hypothesis set $ \small{H} $:
    – can contain good or bad hypotheses
    – up to $ A $ to pick the ‘best’ one as $ g $

learning model = A and H

machine learning:
use data to compute hypothesis g that approximates target f.

Machine Learning and Other Fields

Data Mining

use (huge) data to find property that is interesting.

  • if ‘interesting property’ same as ‘hypothesis that approximate target’
    ML = DM
  • if ‘interesting property’ related to ‘hypothesis that approximate target’
    DM can help ML, and vice versa (often, but not always)
  • traditional DM also focuses on efficient computation in large database

difficult to distinguish ML and DM in reality.

Artificial Intelligence

compute something that shows intelligent behavior.

  • $ \small{g \approx f} $ is something that shows intelligent behavior
    ML an realize AI, among other routes
  • e.g. chess playing
    – traditional AI: game tree
    – ML for AI: ‘learning from board data’

ML is one possible route to realize AI

Statistics

use data to make inference about an unknown process

  • $ g $ is an inference outcome; $ \small{f} $ is something unknown, Statistics can be used to achieve ML
  • traditional statistics also focus on provable results with math assumptions , and care less about computation

statistics: many useful tools for ML

$ \small{E(w) = \frac{1}{2}w^TAw + b^Tw} $

$ \small{dE = \frac{1}{2}dw^TAw + \frac{1}{2}w^TAdw + b^Tdw} \small{\qquad = tr(\frac{1}{2}dw^TAw + \frac{1}{2}w^TAdw + b^Tdw)} \small{\qquad = tr(\frac{1}{2}wTATdw + \frac{1}{2}w^TAdw + b^Tdw)} \small{\qquad = tr(w^TAdw + b^Tdw)} \small{\qquad = tr((w^TA + b^T)dw)} $

$ \therefore \frac{\partial E}{\partial w}^T = \small{w^TA + b^T} \therefore \small{\nabla E(w)} = \frac{\partial E}{\partial w} = \small{A^Tw + b = Aw + b} $

similarly:
$ \small{d} \small{\nabla E(w)} = \small{A dw} \therefore \small{\nabla}^2 E(w) = A $
