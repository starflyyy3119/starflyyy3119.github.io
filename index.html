Shortest Path

Problem: Single Source Single Target Shortest Paths

Goal: Find the shortest paths from source vertex s to some target vertex t.

Observation: Solution will always be a path with no cycles (assuming non-negative weights).

Goal: Find the shortest paths from source vertex s to every other vertex.

Trickier observation: Solution will always be a tree.

  • Can think of as the union of the shortest paths to all vertices.

Why is the solution a tree:

  • Can’t include cycles (no reason to go in a loop)
  • Every node has one parent.

If a graph has no cycles and no nodes has two parents, it’s a tree.

  • If one path is longer, it shouldn’t be part of the solution.
  • If they’re the same, it’s ok to pick one arbitrarily.

SPT Edge Count

If G is a connected edge-weighted graph with V vertices and E edges, how many edges are in the Shortest Paths Tree (SPT) of G?(assume every vertex is reachable)

  • ans: V - 1, because for each vertex, there is exactly one input edge.

Finding a Shortest Path Tree Algorithmically

Dijkstra’s Algorithm.

  • Visit vertices in order of best-known distance from source, relaxing each edge from the visited vertex (relax an edge: Add to SPT if better distance)

Why is Dijkstra’s correct for non-negative edges? Path to X is optimal after X has been dequeued. Inductive argument (proof sketch):

  • Suppose path to just-dequeued vertex v is optimal. Then after relaxation of v’s edges, path to vertex X at top of PQ will be optimal.

Dijkstra’s Algorithm Runtime

Priority Queue operation count, assuming binary heap based PQ:

  • Insertion: V, each costing O(log V) time.
  • Delete-min: V, each costing O(log V) time.
  • decreasePriority: E, each costing O(logV) time.

Overall runtime: O(Vlog(V) + Vlog(V) + ElogV).

  • Assuming E > V, this is just O(ElogV).

The Problem with Dijkstra’s

Dijkstra’s will explore every vertices.
In some situation, we have only a single target in mind, so we need a different algorithm.
So we use some heuristic methods, like A*.

原文来源

LeetCode 问题

  • 121 Best Time to Buy and Sell Stock
  • 122 Best Time to Buy and Sell Stock II
  • 123 Best Time to Buy and Sell Stock III
  • 188 Best Time to Buy and Sell Stock IV
  • 309 Best Time to Buy and Sell Stock with Cooldown
  • 714 Best Time to Buy and Sell Stock with Transaction Fee

框架

核心问题: 给定一个 prices 数组, 什么决定了我们的最大收益？

  • 显式的因素有哪一天、我们可以进行 几次 交易。
  • 隐式的因素是我们当前手上是否有股票。

Notations:

  • prices: 长度为 n 股票价格数组。
  • i: 代表第 i 天。其中 i $ \in $ [0, n-1]。
  • k: 代表允许进行的最大交易次数。
  • dp[i][k][0]: 代表在第 i 天结束时，最多进行 k 次交易，我们手上没有股票，所能获得的最大收益。
  • dp[i][k][1]: 代表在第 i 天结束时，最多进行 k 次交易， 我们手上有股票，所能获得的最大收益。

Base cases:

  • dp[-1][k][0] = 0. 表示没有进入交易日，那么没有收益。
  • dp[i][0][0] = 0. 表示在第 i 天, 最多进行 0 次交易，也就是说没有交易，那么没有收益。
  • dp[-1][k][1] = -Inf. 表示没有进入交易日，手上却有股票，这不可能，用 -Inf 表示这种不可能。
  • dp[i][0][1] = -Inf, 表示没有交易，手上却有股票，这不可能。

状态转移:

  • dp[i][k][0] = max(dp[i-1][k][0], dp[i-1][k][1] + prices[i])
  • dp[i][k][1] = max(dp[i-1][k][1], dp[i-1][k-1][0] - prices

解释： 第 i 天结束时，手上没有股票， 那么可能是第 i - 1 天也没有股票，然后采取了 rest 的策略， 或者是第 i - 1 天有股票然后在第 i 天采取了 sell 的策略； 第 i 天结束时， 手上有股票， 那么可能是第 i - 1 天手上就有股票， 然后采取了 rest 的策略，也可能是第 i - 1 天手上没有股票，采取了 buy 的策略。

应用

Case I: k = 1

也就是只允许进行一次交易。
对于第 i 天，我们有两种状态：

  • dp[i][1][0] = max(dp[i-1][1][0], dp[i-1][1][1] + prices[i])
  • dp[i][1][1] = max(dp[i-1][1][1], dp[i-1][0][0] - prices[i]) = max(dp[i-1][1][1], -prices[i])

我们发现第 i 天的操作只与第 i - 1 天有关，可以优化到 O(1).

再理解一下 k 的意义，k 是允许进行的最大交易次数，dp[i][k] 表示到第 i 天结束时，最多进行了 k 次交易， 也可以比 k 次少。

如，在上题中，我们有 prices = [7, 6, 5, 4, 3, 2], 从开始一直到最后 dp_i10 恒等于0，一次交易都没有做。

Case II: k = +Inf

也就是交易次数不限。
这时 k + 1 与 k 没有区别(因为 k 是正无穷)

  • dp[i][k][0] = max(dp[i-1][k][0], dp[i-1][k][1] + prices[i])
  • dp[i][k][1] = max(dp[i-1][k][1], dp[i-1][k-1][0] - prices[i]) = max(dp[i-1][k][1], dp[i-1][k][0] - prices[i])

Case III: k = 2

限制最多交易两次。

  • dp[i][2][0] = max(dp[i-1][2][0], dp[i-1][2][1] + prices[i])
  • dp[i][2][1] = max(dp[i-1][2][1], dp[i-1][1][0] - prices[i]
  • dp[i][1][0] = max(dp[i-1][1][0], dp[i-1][1][1] + prices[i]
  • dp[i][1][1] = max(dp[i-1][1][1], dp[i-1][0][0] - prices[i]) = max(dp[i-1][1][1], -prices[i])

Case IV: k is arbitrary

限制最多交易 k 次。
注意到进行一次交易需要2天来完成，所以当 k > prices.length >>> 1 时，k 就没有了限制，相当于 caseII.

Case V: k = +Inf but with cooldown

冷冻期为一天，也就是说卖出股票后，无法在第二天买入。
需要将 Case II 稍作改动。

Case II:

  • dp[i][k][0] = max(dp[i-1][k][0], dp[i-1][k][1] + prices[i])
  • dp[i][k][1] = max(dp[i-1][k][1], dp[i-1][k-1][0] - prices[i]) = max(dp[i-1][k][1], dp[i-1][k][0] - prices[i])

如果想在第 i 天买入，那么第 i - 2 天需要保证没有股票，否则将会违反规则。
所以，修改后的状态转移方程为：

  • dp[i][k][0] = max(dp[i-1][k][0], dp[i-1][k][1] + prices[i])
  • dp[i][k][1] = max(dp[i-1][k][1], dp[i-2][k][0] - prices[i])

Case VI: k = +Inf but with transaction fee

选择在买入的时候减去手续费(也可以选择在卖出的时候减去)，大体上与 Case II 如出一辙。

Tree Traversal

Level order

  • Traverse top-to-bottom, left-to-right
  • We say that the nodes are ‘visited’ in the given order.

Depth First Traversals

  • Preorder, Inorder, Postorder
  • Basic(rough) idea: Traverse “deep nodes” before shallow ones.

Depth First Traversals

A Weird Trick

We walk the graph, from top going counter-clockwise.

  • Preorder traversal: Shout every time we pass the LEFT of a node.
  • Inorder traversal: Shout when you cross the bottom.
  • Postorder traversal: Shout when you cross the right.

Visitor Pattern

When writing general tree traversal code. Avoid rewriting traversal for every task of interest(print, sum filesizes, etc) by using the Visitor pattern.

Level Order Traversal

Tree Height

For algorithms whose runtime depends on height, difference between bushy tree and spindly tree can be huge!

Iterative deepening runtimes: $\Theta(N) vs. \Theta(N^2)$

First One:

  • Level 0: 1
  • Level 1: 1 + 2 = 3
  • Level 2: 1 + 2 + 4 = 7
    …
  • Level H: 1 + 2 + 4 + … + $ 2^H $ = $ 2^{H+1} - 1 $

Total = $ 2^1 + 2^2 + … + 2^{H+1} - (H + 1) $
And we know that $ H = \Theta(logN) $
Therefore: The Iterative deepening runtimes = $ \Theta(N) $

Second One:

  • Level 0: 1
  • Level 1: 1 + 1 = 2
  • Level 2: 1 + 1 + 1 = 3
    …
  • Level H: 1 + 1 + … + 1 = H

Total = 1 + 2 + 3 + … H
And $ H = \Theta(N) $
Therefore: The Iterative deepening runtimes = $ \Theta(N^2) $

Geometric Search

Suppose we want an operation that returns all items in a range:

Pruning: Restricting our search to only nodes that might contain the answers we seek.

Runtime for our search: $ \Theta(logN + R) $

  • N: Total number of items in tree.
  • R: Number of matches.

Handling Multidimensional Data: Quadtrees

Quadtrees:

  • Divide and conquer by splitting 2D space into four quadrants.
    — Store items into appropriate quadrant.
    — Repeat recursively if more than one item in a quadrant.

Definition, quadtree is either:

  • Empty
  • A ‘root’ item at some position (x, y) AND four quadtrees that are northwest, northwest, southwest, southeast of (x, y)
  • Use TWO compares to decide which direction to go.

Priority Queue

The Priority Queue Interface

Useful if you want to keep track of the “smallest”, “largest”, “best” etc. seen so far.

How Would We Implement a MinPQ

Some possiblities:

  • Ordered Array
  • Bushy BST: Maintaining bushiness is annoying. Handling duplicate priorities is awkward.
  • HashTable: No good! Items go into random places.
Ordered Array Bushy BST Hash Table Heap
add $ \Theta(N) $ $ \Theta(logN) $ $ \Theta(1) $
getSmallest $ \Theta(1) $ $ \Theta(logN) $ $ \Theta(N) $
removeSmallest $ \Theta(N) $ $ \Theta(logN) $ $ \Theta(N) $

Introducing the Heap

BSTs would work, but need to be kept bushy and duplicates are awkward.

Binary min-heap: Binary tree that is complete and obeys min-heap property.

  • Min-heap: Every node is less than or equal to both of its children.
  • Complete: Missing items only at the bottom level (if any), all nodes are as far left as possible.

What Good Are Heaps?

Heaps lend themselves very naturally to implementation of a priority queue.

  • getSmallest(): return the item in the root node.
  • add(x): place the new employee in the last position, and promote as high as possible.
  • removeSmallest(): assassinate the president (of the company), promote the rightmost person in the company to president. The demote repeatedly, always taking the ‘better’ successor.

Tree representations

General approach: Create mapping from node to children.

Sibling Tree:


Approach 2: Store keys in an array. Store parentID in an array.

  • Similar to what we did with disjointSets.

Approach 3: Store keys in an array. Don’t store structure anywhere.

  • To interpret array: Simply assume tree is complete.
  • Obviously only works for “complete” trees.

Real implementation

Store keys in an array. Offset everything by 1 spot.

  • Makes computation of children/parents “nicer”.
    — leftChild(k) = k 2
    — rightChild(k) = k     2 + 1
    — parent(k) = k / 2

getSmallest $ \Theta(1) $ $ \Theta(logN) $ $ \Theta(N) $ $ \Theta(1) $
removeSmallest $ \Theta(N) $ $ \Theta(logN) $ $ \Theta(N) $ $ \Theta(logN) $

Notes:

  • Heap is $ logN $ time AMORTIZED (some resizes, but no big deal).
  • BST can have constant getSmallest if you keep a pointer to smallest.
  • Heaps handle duplicate priorities much more naturally than BSTs.
  • Array based heaps take less memory.

集合框架

  • 集合框架是一个用来代表和操纵集合的统一架构。
  • 整个集合框架围绕一组标准接口设计。这些接口存在标准实现。如：LinkedList, HashSet, 和 TreeSet 等。除此，可以通过这些接口实现自己的集合。
  • Java 集合框架主要包括两种类型的容器，一种是集合 (Collection), 储存一个元素集合; 另一种是图 (Map), 存储键/值对映射。
  • 所有的集合框架包含以下内容：
    — 接口： 是代表集合的抽象数据类型。如：Collection、List、Set、Queue、Map 等。定义多个接口，是为了以不同的方式操纵集合对象。
    — 实现(类)： 是集合接口的具体实现。它们可以重复使用的数据结构，如： ArrayList、LinkedList、HashSet、HashMap.
    — 算法： 是实现集合接口的对象里的方法执行的一些有用的计算，如：搜索和排序。

  • Java 集合框架位于 java.util 包中，使用集合框架时需要导入包。

Note: 数组和集合的区别

  • 数组长度不可变化而且无法保存具有映射关系的数据；集合类用于保存数量不确定的数据，以及保存具有映射关系的数据。
  • 数组元素既可以是基本类型的值，也可以是对象; 集合只能保存对象。

集合接口

接口 接口描述
Collection 最基本的集合接口，一个 Collection 代表一组 Object。 Java 不提供直接继承自 Collection 的类， 只提供继承于的子接口(如 List 和 Set)。Collection 接口储存一组不唯一、无序的对象。
List List接口是一个有序的 Collection，使用此接口能精确的控制每个元素插入的位置，能够通过索引（元素在 List 中位置，类似于数组的下标）来访问 List 中的元素，第一个元素的索引为 0，而且允许有相同的元素。List 接口存储一组不唯一，有序(插入顺序)的对象。
Set Set接口存储一组唯一，无序的对象。
Queue Queue接口是一种先进先出的数据结构，并提供插入、提取、检验等操作。
Map Map 接口存储一组键值对象，提供 key(键) 到 value(值) 的映射

Note: Set 和 List 的区别

  • Set 接口实例存储是无序的，不重复的数据。List 接口实例存储的是有序的，可以重复的元素。
  • Set 检索效率低下，删除和插入效率高，插入和删除不会引起元素位置改变，实现类有 HashSet， TreeSet。
  • List 和数组类似，但可以动态增长。根据实际存储的数据的长度自动增长 List 的长度。查找元素效率高，插入删除效率低，因为会引起其他元素的位置改变，实现类有 ArrayList，LinkedList， Vector。

集合实现类（集合类）

Java提供了一套实现了集合接口的标准集合类。其中一些是具体类，这些类可以直接使用，另外一些是抽象类，提供了接口的部分实现。
|集合实现类|特点|
|-|-|
|ArrayList|实现了List接口。使用数组存储元素，数组可以动态创建，如果元素个数超过了数组的容量，那么就增长当前长度的50%，并将当前数组中的所有元素都复制到新的数组中。该类非同步，在多线程的情况下不要使用。插入删除效率低。|
|LinkedList|实现了List接口。允许有 null 元素。主要用于创建链表数据结构。该类没有同步方法，如果多个线程同时访问一个 List，必须自己实现访问同步。|

|TreeSet|实现了Set接口，可以实现排序等功能。|
|HashMap|实现了Map接口。HashMap是一个散列表，它存储的内容是键值对(key-value) 映射。根据键的 HashCode 值存储数据，具有很快的访问速度，最多允许一条记录为null， 不支持线程同步。在 JDK1.8 中， HashMap 采用数组 + 链表 + 红黑树实现，当链表长度超过阈值(8) 时，将链表转化成红黑树。|

Note: 在实际使用中，如果更新图时不需要保持图中元素的顺序，就使用 HashMap，如果需要保持图中元素插入顺序或者访问顺序，就使用 LinkedHashMap, 如果需要按照键值排序，就使用 TreeMap。
